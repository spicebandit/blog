# 블로그 자동 발행 가이드 (Claude Code용)

이 저장소는 Astro 정적 블로그다. 글은 `src/content/blog/*.md`에 저장되며,
main 브랜치에 push하면 Vercel이 자동 배포한다.

## 글 작성 규칙 (/blog 커맨드 또는 "OO 주제로 글 써줘" 요청 시)

1. 파일명: 영문 소문자 + 하이픈 슬러그 (예: `ai-automation-guide.md`) — URL이 되므로 SEO 키워드 포함
2. frontmatter 필수 항목:
   - `title`: 타겟 키워드를 앞쪽에 포함, 30자 내외
   - `description`: 80~150자 메타 디스크립션. 검색결과에 노출되는 문장이므로 클릭을 유도하는 요약
   - `pubDate`: 발행 시각을 시간까지 ISO로 (예: `2026-06-13T14:30:00+09:00`). 목록은 `pubDate` 내림차순 정렬이라, **같은 날 글이 둘 이상이면 시간이 있어야 최신 글이 맨 위로** 온다. 날짜만(YYYY-MM-DD) 쓰면 같은 날끼리 순서가 섞인다.
     - **★ 발행일 = 실제 발행 시점 (작성일 아님)**: 블로그에 표시되는 날짜는 `pubDate`다. **draft를 제거하고 발행하는 그 순간의 실제 날짜·시각으로 `pubDate`를 반드시 갱신**한다. 초안을 며칠 전에 써뒀더라도 작성 당시 날짜를 그대로 두지 말 것 — 독자에게는 "오늘 발행"인데 며칠 전 날짜로 보이면 안 된다. 시리즈(①②③④)를 같은 날 여러 편 발행하면, 날짜는 모두 오늘로 두고 **시각(HH:MM)만 순서대로** 다르게 줘서 정렬을 맞춘다(과거 날짜로 backdate해서 시리즈 순서를 맞추지 않는다).
   - `category`: **필수**. `energy` / `economy` / `ai` / `life` 중 정확히 하나(영문 슬러그). 4개 외 값이거나 누락 시 빌드가 실패한다. 카테고리 정의는 `src/lib/categories.ts` 한 곳에서 관리한다.
     - 영역 ↔ 카테고리 매핑: 에너지 글 → `energy`, 경제·경영 글 → `economy`, AI·AX 글 → `ai`, Editor 글 → `life`
   - `tags`: 2~4개
3. 본문 SEO 규칙:
   - h2(`##`) 소제목 3개 이상, 소제목에도 연관 키워드 자연스럽게 포함
   - 분량 1,500자 이상 (정보성 글 기준)
   - 첫 문단에 핵심 키워드 포함, 질문에 바로 답하는 구조
   - 과장/허위 정보 금지, 출처가 필요한 수치는 명시
4. 글쓰기 품질 규칙 (정보성 글 = 에너지/경제/AI):
   - **구체성**: 추상적 설명에는 가능하면 구체적 예시나 숫자를 하나씩 붙인다. (예: "디지털 상품은 회수율이 높다"에 그치지 말고 "전자책은 한 번 만들면 추가 비용 없이 1,000번 팔 수 있다"처럼.) 단, 지어낸 수치는 금지 — 일반적으로 알려진 사실이나 예시 수준으로.
   - **관점**: 교과서적 나열을 피하고, 글쓴이의 관점이나 경험이 묻어나는 문장을 한두 개 넣는다. 누가 써도 똑같은 글이 아니라 한 사람의 시각이 느껴지게.
   - **하나의 레슨**: 글 전체가 전하려는 단 하나의 핵심 메시지를 분명히 정하고, 독자가 다 읽고 나서 그 한 가지가 잔상에 남도록 마무리한다. 억지 요약이 아니라, 글의 흐름이 자연스럽게 그 한 가지로 수렴하게.
   - **정의와 맥락**: 핵심 주제에 대한 간단한 정의와 역사적 흐름을 글 초반에 짚는다. "이것이 무엇인지(정의)"와 "어떻게 여기까지 왔는지(흐름·배경)"를 간결하게 보여 독자가 맥락을 잡게 한 뒤 본론으로 들어간다. 단, 백과사전식 나열이 아니라 글의 핵심 메시지로 자연스럽게 이어지는 도입부 역할을 하도록 하고, 분량은 한 문단 정도로 과하지 않게 한다. (life 글에는 적용하지 않음)
5. 발행:
   ```bash
   git add . && git commit -m "post: <글 제목>" && git push
   ```
6. 발행 후 사용자에게 글 URL 예상 경로(`/blog/<슬러그>/`)를 알려줄 것

## 이미지 삽입 규칙

모든 글에는 본문 흐름에 맞춰 이미지 2~3개를 삽입한다.

1. 이미지 가져오기: 글 작성 시 `scripts/fetch-images.mjs`를 사용해 주제에 맞는 **영문 키워드**로 이미지를 가져온다.
   ```bash
   node scripts/fetch-images.mjs "<영문 키워드>" 3                      # 기본 소스: unsplash
   node scripts/fetch-images.mjs "<영문 키워드>" 3 --source=wikimedia   # 위키미디어 커먼스
   ```
   출력된 마크다운 스니펫(이미지 + 출처)을 본문에 그대로 붙여 쓴다.
   - **소스 선택 기준**: 분위기·개념용 스톡사진은 `unsplash`(기본), 실제 인물·기업·시설·지도·차트 등 *주제 직결* 이미지는 `--source=wikimedia`. 시사성 강한 글은 둘을 섞어 써도 된다.
2. 배치: 이미지는 소제목(h2) 근처에 자연스럽게 배치한다. 한 곳에 몰지 말고 글 전체에 분산한다.
3. 출처 표기(필수): 각 이미지 바로 아래에 스크립트가 만들어 준 출처 문구를 **그대로** 넣는다(소스별 라이선스 약관에 맞춰 자동 생성되므로 임의로 바꾸지 않는다).
   - Unsplash: `*Photo by [작가명](프로필링크) on [Unsplash](사진링크)*`
   - Wikimedia: `*이미지: 작가명 · [Wikimedia Commons](파일페이지) · [라이선스](라이선스링크)*` (CC/PD 라이선스 준수 — 작가·출처·라이선스 표기 필수)
4. 이미지 선택 기준: 정보성 글은 주제와 직접 관련된 이미지를, 가벼운 글은 분위기에 맞는 이미지를 고른다.
5. 이미지는 **빌드 시점 고정 방식**이다. 글 작성 시점에 미리 가져와 마크다운에 URL을 직접 박으며, 방문자가 페이지를 열 때 Unsplash API를 호출하지 않는다.

## 데이터·근거 표시 규칙 (필수 — 편집국장 검수 항목)

기사에서 수치, 실적, 통계, 법령·정책 변경이 언급될 때는 반드시 구조화된 형태로 근거를 제시해야 한다. 이 규칙을 충족하지 않은 기사는 발행 불가다.

### 1. 실적·수치 데이터가 있는 기사

기업 실적, 경제지표, 시장 데이터 등 수치가 핵심인 기사:

- 본문에 마크다운 표(`|`로 구분)를 삽입하여 핵심 지표를 나열한다
- 표에는 **비교 기준 기간, 전분기 대비(QoQ), 전년 대비(YoY) 증감률**을 반드시 포함한다
- 출처(기업 IR 자료, 공시, 뉴스 URL 등)를 표 바로 아래에 표기한다
- 수치 단위를 명확히(억 원, B달러, % 등) 기재한다

**형식 예시 (기업 실적)**:

| 항목 | 당분기 | 전분기 | QoQ | 전년동기 | YoY |
|------|--------|--------|-----|----------|-----|
| 매출 | $8.05B | $7.75B | +3.9% | $5.82B | +38.3% |
| 영업이익 | $1.55B | $1.17B | +32.5% | -$1.26B | 흑자전환 |
| 순이익 | $1.58B | $1.10B | +43.6% | -$1.23B | 흑자전환 |

*출처: Micron Technology IR 보도자료 (2025-03-19), [링크]*

### 2. 법령·정책·규정이 변경된 기사

법령·규정·정책이 신설·개정·폐지된 경우:

- **변경 전**과 **변경 후** 조문을 원문 그대로 블록쿼트(`>`)로 인용한다
- 핵심 변경 부분은 **굵게** 처리하여 시각적으로 구분한다
- 시행일, 법률(시행령) 번호, 출처(국가법령정보센터 URL 또는 관보)를 명시한다
- 조문 전체 인용이 과도하면 해당 조항·항·호만 잘라서 인용하되, "…(중략)…"로 생략 표시

**형식 예시 (법령 개정)**:

**변경 전 (구 조문)**
> 제65조(전기요금의 인가) 전기사업자는 전기요금 및 그 밖의 공급조건을 정하여 산업통상자원부장관의 **인가**를 받아야 한다.

**변경 후 (개정 조문)**
> 제65조(전기요금의 인가) 전기사업자는 전기요금 및 그 밖의 공급조건을 정하여 산업통상자원부장관에게 **신고**하여야 한다.

*출처: 전기사업법 일부개정 (법률 제○○호, 시행 2025. 1. 1.), 국가법령정보센터*

### 3. 편집국장 검수 체크리스트

기사 발행 전 아래 항목을 확인한다:
- [ ] 수치가 있는 경우 → 표로 정리되어 있는가?
- [ ] 표에 QoQ·YoY 또는 적절한 비교 기간이 포함되어 있는가?
- [ ] 법령·정책 변경 기사 → 원문 조문이 인용되어 있는가?
- [ ] 모든 표·인용문에 출처가 명시되어 있는가?
- [ ] 출처 링크(URL)가 유효한가?

## 안전 규칙 (모든 글에 적용 — 반드시 준수)

- **에너지 관련 글**: 공개된 일반적 시장 동향·정책·기술 정보만 다룬다. 특정 회사의 내부 정보, 비공개 사업 내용, 미공개 실적은 절대 다루지 않는다.
- **경제/주식 관련 글**: 특정 종목 매수·매도 추천을 하지 않는다. 시장 동향, 경제지표 해설, 역사적 배경과 일반적 트렌드 분석 위주로 쓰고, "투자 조언이 아니다"라는 점이 글에 드러나게 한다.
- **모든 글**: 확인되지 않은 사실, 과장, 허위 정보를 쓰지 않는다. 불확실한 수치는 단정하지 않는다.

## 매일 자동 초안 생성 (반자동)

> **[2026-06-22 시스템 전환 후]** Paperclip 언론사 시스템 도입으로 **콘텐츠 담당이 EditorInChief 에이전트로 이관**됐다. 아래 흐름(Claude Code 직접 작성)은 Paperclip 없이 Claude Code만 쓸 때의 fallback 경로다. Paperclip 시스템에서는 EditorInChief가 주제 선정·초안 작성을 담당하며, CTO는 플랫폼(코드·배포·애널리틱스)만 담당한다.

매일 초안을 만들되 **발행은 자동으로 하지 않는다**. draft까지만 만들고, 사람이 검토 후 `draft: true`를 지우고 발행한다. 글 작성은 **외부 API를 호출하지 않고 Claude Code가 직접** 한다(추가 API 비용 없음).

**역할 분리**
- `scripts/daily-draft.mjs` (순수 Node, API 호출 없음): 오늘 요일 → 영역 결정 → 영역별 주제 풀에서 날짜 기반으로 키워드 후보 3개 선정 + 1개 추천. 그게 전부다(글은 쓰지 않음).
  - `node scripts/daily-draft.mjs` (사람용 출력) / `--json` (자동화용)
  - 같은 날 재실행하면 같은 결과(idempotent)
- **Claude Code (Paperclip 없을 때)**: 스크립트가 추천한 키워드를 받되, **시사 이슈를 함께 검토해 최종 키워드를 정한 뒤** 위 글 작성/SEO/안전/이미지 규칙대로 초안 md를 작성한다.
  - **뉴스 검토(주제 선정 단계)**: 글을 쓰기 전, 웹 검색으로 **뉴욕타임즈(NYT)와 국내 주요 뉴스**(연합뉴스·네이버 뉴스 등)에서 **오늘 영역(요일별 로테이션)과 관련해 지금 이슈가 되는 주제**를 확인한다.
    - 반영 방식은 **"영역 안에서 키워드만 조정"** — 요일별 영역(예: 토=ai)은 그대로 두고, 그 영역 안에서 뉴스에 뜨는 이슈를 우선해 최종 키워드를 고른다. 뉴스 이슈가 영역에 더 잘 맞으면 스크립트가 제안한 풀의 후보를 벗어나 그 키워드로 써도 된다(영역 자체는 바꾸지 않는다).
    - 시의성 있는 뉴스가 마땅치 않으면 스크립트의 추천 키워드를 그대로 쓴다.
    - 뉴스는 **맥락·트렌드 참고용**이다. 안전 규칙(특정 종목 추천 금지, 내부정보 금지, 미확인 사실 금지)은 그대로 지키고, 속보성 단정 대신 일반적 흐름·배경 해설로 풀어낸다.
  - 파일: 스크립트가 제안한 `src/content/blog/<날짜>-<슬러그>.md` (최종 키워드에 맞춰 슬러그는 조정 가능)
  - frontmatter에 `category`(오늘 영역)와 `draft: true`를 넣는다
  - 이미지는 `scripts/fetch-images.mjs`로 받아 삽입

**요일별 영역 로테이션**: 월=energy, 화=economy, 수=ai, 목=energy, 금=economy, 토=ai, 일=life

**매일 자동화하려면**: Claude Code 스케줄(/schedule 루틴 등)로 매일 1회 다음을 실행 — "`node scripts/daily-draft.mjs --json` 실행 → **NYT·국내 주요 뉴스에서 오늘 영역 관련 이슈를 웹 검색으로 검토 → 영역 안에서 최종 키워드 확정** → CLAUDE.md 규칙대로 초안을 `draft: true`로 작성 → push하지 말고 알림만". (cron으로 스크립트만 돌리면 키워드만 정해지고 글은 안 써지므로, 뉴스 검토·글 작성 단계는 Claude Code가 맡아야 한다.)

**검토**: draft 글은 빌드/목록/카테고리/사이트에 노출되지 않는다(`draft` 필터). 내용 확인 후 발행한다.

## 글 작성·발행 후 옵시디언 동기화

새 글을 **발행한 뒤(= `draft` 제거하고 git push 한 뒤)** 반드시 다음을 실행해 옵시디언 보관함에 동기화한다.

```bash
node scripts/sync-to-obsidian.mjs
```

- 동작: 발행된(draft 아님) 모든 글을 옵시디언 보관함의 **`Blog/` 전용 폴더**(`~/Documents/obsidian/mynotes/Blog/`)에 노트로 등록한다. idempotent하므로 매번 전체를 다시 써도 안전하다(인자 없이 전체 실행 권장 — 관련 글 링크가 양방향으로 갱신됨).
- 옵시디언 노트: frontmatter(title/category/tags/date/url) + 본문 + 라이브 블로그 URL + **같은 카테고리·태그 글과 `[[위키링크]]` 상호 연결**(`## 관련 글` 섹션).
- 충돌 방지: 보관함의 `Blog/` 폴더에만 쓴다(사용자가 직접 만든 다른 노트는 건드리지 않음). 라이브 데이터 페이지(예: 월드컵 트래커)는 `SKIP_SLUGS`로 제외한다.
- draft 글은 동기화하지 않는다(발행된 글만). 예약 발행 스크립트에는 이 동기화가 이미 포함돼 있다.

## 절대 하지 말 것
- draft가 아닌 기존 글 무단 수정/삭제
- astro.config.mjs의 site 값 변경 (사용자 승인 필요)
- `.env`(UNSPLASH_ACCESS_KEY 등 시크릿) 커밋 — `.gitignore`로 제외되어 있음
- 초안(draft) 글을 사람 검토 없이 자동 발행
